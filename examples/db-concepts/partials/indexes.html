Indexes trade write cost for read speed

Without an index, finding a row means reading the table. With a B-tree index, the engine walks a sorted tree and jumps near the match — often orders of magnitude fewer pages.

access path

Rule of thumb: index columns you filter/join on a lot (WHERE, JOIN … ON, sometimes ORDER BY). Don’t index everything — each index slows inserts/updates and uses disk.

Composite indexes

Order matters. (user_id, created_at) helps “orders for user X, newest first” but won’t help a bare WHERE created_at > … as well.
